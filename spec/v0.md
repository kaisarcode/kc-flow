# kc-stdio Spec v0

`kc-stdio` is a headless-first contract and flow engine.

The source of truth is the executable contract model plus the composed flow
model interpreted by the engine.

## Core Rule

A tool and a workflow are the same kind of unit.

They are used in two modes:

- standalone: opened and edited directly
- embedded: imported and represented as a tool inside another flow

The base unit is the contract.

## Model Layers

`kc-stdio` is split into these layers:

- `contract`: executable unit definition
- `flow`: composition of contracts through nodes and links
- `engine`: headless resolution and execution
- `cli`: non-GUI interface to inspect and run
- `gui`: optional visual editor over the same model

## Contract

A contract defines one executable unit.

A contract may be:

- atomic: backed directly by one script or command
- composed: backed by an internal flow of other contracts

Every contract is defined through:

- identity
- public interface
- runtime implementation
- binding rules
- optional internal composition

## Contract Identity

Minimum identity fields:

- `id`
- `name`
- `version`
- `description`

## Public Interface

A contract may expose three public interface sections:

- `params`
- `inputs`
- `outputs`

A contract may expose:

- a full public interface
- a partial public interface
- a param-only interface

### Params

`params` are editable configuration values.

They may be set directly in the current contract instance or mapped from a
parent flow.

Typical examples:

- width
- height
- model path
- runner flags
- prompt template

Minimum param fields:

- `id`
- `type`
- `required`
- `default`
- `label`
- `description`

### Inputs

`inputs` are external values that may be connected from another contract or
provided directly by the parent flow.

Minimum input fields:

- `id`
- `type`
- `required`
- `label`
- `description`

### Outputs

`outputs` are values exposed by the contract to other contracts or to the
parent flow.

Minimum output fields:

- `id`
- `type`
- `label`
- `description`

## Port Types

`kc-stdio` treats everything as input/output through explicit value flow.

The interface describes semantic value types together with transport
materialization.

Initial useful types:

- `text`
- `int`
- `float`
- `bool`
- `path`
- `file`
- `file.image`
- `dir`
- `json`
- `bytes`
- `stream`
- `exit_code`

## Runtime

The runtime section belongs to the contract.

It defines how the contract is actually executed.

Initial runtime fields:

- `runner`
- `script`
- `workdir`
- `env`

Examples of runner values:

- `sh`
- `bat`
- `cmd`
- `powershell`
- `php`
- `binary`

`kc-stdio` executes the runner declared by the contract runtime.

## Bindings

Bindings define how public params and inputs are materialized into the
runtime and how outputs are extracted back from execution.

Bindings connect public interface fields to concrete execution mechanisms.

Initial input binding modes:

- `arg`
- `env`
- `stdin`
- `file`
- `expr`

Initial output binding modes:

- `stdout`
- `stderr`
- `file`
- `exit_code`

Each binding maps a contract interface field to one execution mechanism.

## Flow

A flow is a composed contract graph.

A flow contains:

- `nodes`
- `links`
- `expose`

### Nodes

A node is one instance of another contract.

A node must define:

- `id`
- `contract`

Optional node state:

- param overrides
- display metadata
- layout metadata

### Links

A link connects one source to one destination.

Minimum link fields:

- `from`
- `to`

Examples:

- parent input to child input
- child output to sibling input
- child output to sibling param

### Expose

`expose` maps internal node interface fields to the public interface of the
parent contract.

This is what allows a workflow to be imported as a tool.

## Engine

The engine is headless and executes the contract and flow model directly.

The engine is responsible for:

- loading a contract or flow file
- validating required fields
- resolving links and dependencies
- materializing runtime bindings
- executing nodes in dependency order
- collecting declared outputs

The engine owns loading, validation, resolution, execution, and output
collection.

## GUI Rule

The GUI may represent contracts as nodes, windows, or any other visual form.

The GUI derives its behaviour from:

- contract
- flow
- engine

Every visual operation corresponds to those shared model layers.

## Minimal File Direction

Initial direction:

- one file format for contract definitions
- one file format for composed flow definitions

They may later converge into one format if the distinction becomes
unnecessary.

## Minimal v0 Goal

The first real engine milestone should support:

1. loading one atomic contract file
2. validating params, inputs, outputs, runtime, and bindings
3. running one atomic contract headlessly
4. loading one flow with multiple nodes
5. resolving simple links between node outputs and node inputs
6. exposing flow outputs
